# Pluto Demos — v0.2.6 Insights

This page summarizes what the role-system + collaboration demos in
[`tests/demo_*`](../../tests/) actually showed when run against a real Pluto
server (`127.0.0.1:9200/9201`) and the real GitHub Copilot CLI.

Both demos in this batch (added in `v0.2.6`) exercise:

- The shared **role library** (`library/roles/*.md`, loaded by each agent).
- The shared **coordination protocol** (`library/protocol.md`, inlined into
  the agent prompt by `pluto_agent_friend.py` so the agent works regardless
  of CWD).
- Real Pluto operations: `register`, `agents/send`, `long_poll`,
  `locks/acquire`, `locks/release`, `agents/broadcast`.
- Real `copilot -p ... --model <X> --allow-all-tools --allow-all-paths
  --add-dir <ws> --no-ask-user` invocations.

## 1. [Fractal Collaboration](fractal_collaboration_demo.md)

**Goal.** Show the *full* role pipeline (Orchestrator → Specialist →
Reviewer → QA) on a non-trivial numeric task: render an 800×600 Mandelbrot
PNG and compute two statistics over the grid.

**Result (run on 2026-04-24).**

| metric | value |
|--------|-------|
| total wall-time | 1m 42s |
| tasks completed | 3 / 3 (`t-001`, `t-002`, `t-003`) |
| reviewer verdict | `approved` for all |
| pytest | `0` (3 tests pass) |
| `convergence_ratio` | `0.181` |
| `mean_escape_time` | `5.19` |
| PNG | `/tmp/pluto/demo/fractal_collaboration/outputs/mandelbrot.png` |

**What it actually proved.**

- The role-driven Specialist (real Copilot) implemented all 3 files
  (`mandelbrot.py`, `run_mandelbrot.py`, `test_mandelbrot.py`) in the demo
  workspace, **only** writing files listed in `task.files` and **only**
  while holding the corresponding write lock.
- The Orchestrator never wrote files itself — it only published the task
  list, dispatched `task_assigned` messages, and consumed `task_result` /
  `review` / `qa_result` payloads.
- The lock/release discipline survived contention: every `acquire` returned
  `status: ok` because tasks were correctly serialised by `dependencies`.
- Replaying the trace via `write_report.py` made the agent collaboration
  visible (filtered table in the report).

**Known noise (not blocking).** Pluto's HTTP `long_poll` re-delivered
already-acked messages with empty payloads, producing ~300k `recv
type=None` events over 100s. The report writer now filters these; the
underlying server-side dedup is a follow-up.

## 2. [Haiku Team vs Sonnet Solo](haiku_vs_sonnet_demo.md)

**Goal.** Test the central thesis of the v0.2.6 role design: *can two
weaker models, coordinated through Pluto + roles + reviewer, match or
beat a single stronger model on a non-trivial task?*

**Setup.**

- **Setup A — Haiku Team:** real `claude-haiku-4.5` plays Specialist and
  Reviewer; a Python harness plays Orchestrator and QA. Coordination via
  real Pluto messages and locks. Spec is decomposed into two tasks
  (`t-001` cache class, `t-002` tests) with `dependencies: [t-001]`.
- **Setup B — Sonnet Solo:** one `claude-sonnet-4.5` call receives the
  entire spec and produces both files in one shot. No Pluto, no protocol,
  no reviewer.
- **Same task**, **same QA harness** (`pytest tests/test_lru_ttl.py -q`).

**The task.** Build a single Python class, `LRUTTLCache`, that behaves as
an in-memory key/value cache with three combined properties: bounded size
with LRU eviction, per-entry TTL expiry, and thread-safety. Public API:

```python
class LRUTTLCache:
    def __init__(self, capacity: int, default_ttl: float): ...
    def put(self, key, value, ttl: float | None = None) -> None: ...
    def get(self, key) -> Any | None: ...   # None on miss or expiry
    def __len__(self) -> int: ...           # excludes expired entries
    def clear(self) -> None: ...
```

How each property must work:

- **LRU (Least Recently Used) — capacity-bounded eviction.** The cache
  holds at most `capacity` entries. When `put` is called on a full cache
  with a new key, the entry that was accessed *least recently* is
  evicted to make room. Both `get(key)` (on a hit) and `put(key, value)`
  count as "accessing" the key and must move it to the
  most-recently-used position. The textbook implementation is a
  doubly-linked list of entries plus a `dict` mapping key → list node,
  giving $O(1)$ `get` and `put`; in Python, `collections.OrderedDict`
  with `move_to_end` collapses both into a few lines.
- **TTL (Time-To-Live) — per-entry expiry.** Each entry stores an
  absolute expiry timestamp `expires_at = time.monotonic() + ttl` (using
  `default_ttl` from `__init__` unless `put` overrides it). On every
  `get`, the entry's `expires_at` is compared to `time.monotonic()`; if
  expired, the entry is removed and `get` returns `None` (a miss). TTL is
  checked **lazily on access** — there is no background sweeper thread.
  An expired-but-not-yet-evicted entry must not count toward `__len__` or
  toward the capacity limit when a new `put` arrives.
- **Thread-safety — single coarse lock.** All public methods must be
  safe to call concurrently from multiple threads. A single
  `threading.RLock` (or `Lock`) held for the duration of each public
  method is sufficient and is what the spec asks for; finer-grained
  locking is out of scope. The stress test launches several threads
  hammering `put`/`get` on the same cache and asserts no exception is
  raised and the post-conditions still hold.

The spec lists exactly **8 pytest cases** the implementation must
satisfy: (1) basic `put`/`get` round-trip, (2) capacity-based LRU
eviction picks the right victim, (3) `get` refreshes LRU position so the
just-read key is *not* the next victim, (4) reading an entry past its
TTL returns `None`, (5) an expired entry doesn't consume a capacity slot
when a new key is inserted, (6) `clear()` empties the cache, (7)
`__len__` reflects only live entries, (8) a multi-threaded stress test
that runs without exceptions and leaves the cache in a consistent state.

**Result (run on 2026-04-24, single run).**

| metric | Haiku team (3 calls via Pluto) | Sonnet solo (1 call) |
|--------|-------------------------------:|---------------------:|
| **pytest status** | ✅ `pass` | ❌ `fail` |
| tests passed / failed | **8 / 0** | 12 / 1 |
| copilot calls | 3 | 1 |
| total LLM wall-time | 199.6 s | 152.9 s |
| cache module bytes / lines | 5760 / 176 | 4743 / 145 |
| test file bytes / lines | 4118 / 145 | 5382 / 203 |
| reviewer verdict | `approved` | n/a |

**Why Sonnet solo failed.** It failed
`test_get_refreshes_lru_position` — its `get()` did not refresh the LRU
position correctly under TTL, so after 4 inserts the `.get('a')` came back
`None`. Note Sonnet wrote **more** tests than required (13 vs the 8 in
the spec) but its implementation didn't satisfy one of its own tests.

**Why the Haiku team passed.**

- Decomposition forced the implementation and the tests into separate
  Specialist invocations against the *same* spec, so each invocation could
  focus narrowly. The Specialist for `t-001` thought only about the class;
  the Specialist for `t-002` (a fresh Copilot session) thought only about
  the tests, against the existing class.
- The Reviewer pass (real Haiku) inspected both files against the protocol
  and emitted `verdict: approved` with no findings — i.e. the conservative
  Haiku output happened to be correct.
- Specialist held a **real Pluto write lock** on each file before editing
  (recorded in [`haiku_vs_sonnet/trace.json`](haiku_vs_sonnet/trace.json)).

**Honest caveats.**

- This is a **single run** at v0.2.6. The result is illustrative, not
  statistically robust; rerun N times to estimate variance before quoting.
- The total LLM wall-time was *higher* for the Haiku team (~200s vs ~153s)
  — the collaboration tax is real. The win is correctness, not speed.
- Sonnet's solo prompt deliberately omits the protocol and the role
  definition (that's the experimental contrast). Giving Sonnet the same
  protocol would be a different experiment.

**What this validates for the v0.2.6 design.**

- The role library + protocol document + Specialist single-responsibility
  pattern makes weaker models **noticeably more reliable** on the same
  spec — even when the per-call model is genuinely weaker.
- The Pluto lock discipline is non-disruptive: Specialist still finishes
  in seconds despite the round-trip overhead.
- Inlining `protocol.md` into the role injection (the v0.2.6 fix) is what
  let the Reviewer Haiku produce a structured JSON verdict from any CWD —
  the Reviewer never had to locate `protocol.md` on disk.

## How to re-run

```bash
# Restart Pluto on the demo ports (config in config/pluto_config.json)
./PlutoServer.sh --kill
./PlutoServer.sh --daemon

# Run both demos (real copilot + real Pluto required)
PLUTO_RUN_DEMOS=1 python -m pytest tests/demo_fractal -s
PLUTO_RUN_DEMOS=1 python -m pytest tests/demo_haiku_vs_sonnet -s

# Override the models if needed
PLUTO_HAIKU_MODEL=claude-haiku-4.5 \
PLUTO_SONNET_MODEL=claude-sonnet-4.5 \
PLUTO_RUN_DEMOS=1 python -m pytest tests/demo_haiku_vs_sonnet -s
```
